# Documentation for Local Nodes

*How to capture and share knowledge so your community learns and grows*

---

## Why Documentation Matters

Your community generates knowledge constantly. Decisions are made. Lessons are learned. Processes are created.

Without documentation, this knowledge walks out the door when people leave. When someone asks "how do I...?" you explain it again. When history is needed, it's gone.

With documentation, your community builds institutional memory. New members can catch up. Contributors can build on past work. Your node becomes more than the sum of its current members.

The Greenpill Local Regen Guide emphasizes that communities need to "remember what they've learned" — documentation is how you do that [1].

---

## Types of Documentation

### 1. Governance Documents

These are your rules and agreements:

**Include:**
- Community agreements and values
- Decision-making processes
- Role descriptions and responsibilities
- Voting records and outcomes
- Constitutional documents

**Purpose:** Know how decisions get made, who does what, what rules exist.

### 2. Process Documents

These explain how things work:

**Include:**
- How to join the community
- How to contribute
- Event planning process
- Onboarding steps for new members
- How to propose initiatives

**Purpose:** Enable people to actually do things.

### 3. Historical Documents

These record what happened:

**Include:**
- Meeting notes
- Event recaps and outcomes
- Milestone achievements
- Lessons learned
- Membership changes

**Purpose:** Build institutional memory, avoid repeating mistakes.

### 4. Knowledge Base

This is your reference library:

**Include:**
- FAQs
- Tool guides and tutorials
- Resource lists
- Best practices
- External references

**Purpose:** Answer questions, share expertise.

## What to Document: Priority Guide

### Must-Have (Essential)

If you document nothing else, document these:

1. **How to join** — Clear steps to become part of the community
2. **Community agreements** — Rules and expectations
3. **Contact information** — Who's who, how to reach them
4. **Key resources** — Where to find important things

### Should-Have (Important)

These make your community run better:

1. **Meeting notes** — What was decided, who committed to what
2. **Event archives** — What worked, what didn't
3. **Decision history** — Why choices were made
4. **Role definitions** — What's expected of each position
5. **Financial transparency** — How money is handled

### Nice-to-Have (Helpful)

These add depth:

1. **Detailed processes** — Step-by-step for complex tasks
2. **Templates** — Reusable formats
3. **Training materials** — How to do specific things
4. **Glossary** — Terms and definitions
5. **Onboarding guides** — How to onboard new members

## Maintaining Documentation

Documentation is worthless if outdated. Here's how to keep it current:

### Assign Ownership

Someone needs to be responsible for:
- Keeping docs current
- Organizing new information
- Reviewing for accuracy
- Archiving outdated content

### Review Regularly

Set a schedule:
- **Monthly:** Check for major updates needed
- **Quarterly:** Full review of accuracy
- **Annually:** Restructure and consolidate

### Make It Easy to Update

The easier documentation is to update, the more it gets updated:

- Use familiar tools
- Keep templates simple
- Make edit permissions clear
- Celebrate contributions

## Documentation Workflow

### When Something Happens (Decision/Event)

1. **Capture** — Take notes, record key points
2. **Process** — Summarize and organize
3. **Store** — Put in right place with clear name
4. **Share** — Let relevant people know

### For Ongoing Knowledge

1. **Identify** — What does the community need to know?
2. **Create** — Write clear, accessible documentation
3. **Review** — Check for accuracy
4. **Publish** — Make available in right channels
5. **Update** — Keep current over time

## Best Practices

### Be Clear

Write so someone new can understand:
- Define acronyms
- Explain context
- Use examples

### Be Concise

Respect people's time:
- Get to the point
- Use headers
- Make scannable

### Be Current

Outdated docs cause problems:
- Date everything
- Note when to review
- Remove or mark outdated

### Be Accessible

Make docs findable:
- Clear filenames
- Organized folders
- Good search

---

## Common Mistakes

### 1. Over-Documentation

Don't document everything. Focus on what's needed.

### 2. Outdated Docs

Better to have less documentation that's current than lots of outdated stuff.

### 3. Nobody Owns It

If everyone is responsible, no one is responsible.

### 4. Perfect Is Enemy of Good

Done is better than perfect. Ship it, improve later.
